# Dev 单端口派生与残留实例清理设计

## 背景

当前 `bun run dev` 维护了多组本地开发端口环境变量：

- `DEV_WORKER_PORT`
- `DEV_ATTEMPT_WORKER_PORT`
- `DEV_UI_PORT`

同时，后台运行状态主要依赖 `.dev/dev-runner.json` 中记录的守护进程 PID，并通过该状态推导健康检查目标。实际使用中存在两个问题：

1. 守护进程异常退出、状态文件残留或失真时，`--status` / `--stop` 可能显示“未运行”，但派生子进程或服务端口仍然残留。
2. 多个端口环境变量增加了认知负担，也让“当前这组服务应该占哪些端口”变得不够直观。

## 目标

将本地开发端口收敛为单一公开配置，并增强后台状态检测与停止逻辑，使其能识别并清理残留实例。

## 设计决策

### 1. 端口配置收敛

只保留一个公开环境变量：

- `DEV_PORT`：本地开发主服务端口，默认 `8787`

其余端口不再暴露为环境变量，统一自动派生：

- `worker = DEV_PORT`
- `attempt-worker = DEV_PORT + 1`
- `ui = DEV_PORT + 2`
- `worker inspector = DEV_PORT + 1000`
- `attempt inspector = DEV_PORT + 1001`

选择固定派生而非随机端口，原因是：

- 易于记忆和排查
- 文档与手工调试更简单
- `--status` / `--stop` 可以稳定推导同一组端口

### 2. 后台状态模型增强

`bun run dev -- --status` 不再只依赖状态文件是否存在，也不再把“无状态文件”简单等价为“无后台实例”。状态判断来源扩展为：

1. `.dev/dev-runner.json` 中记录的后台状态
2. 真实运行中的 `scripts/dev.mjs --_daemon` 守护进程
3. 当前 `DEV_PORT` 派生出的已知服务端口是否仍被本项目实例占用

状态分为三类：

- `healthy`：守护进程存在，且健康检查正常
- `degraded`：守护进程存在，但健康检查异常
- `residual`：未识别到有效守护进程，但识别到残留实例或残留端口

其中 `residual` 是本次新增状态，用于明确区分“完全未运行”和“守护进程没了，但现场没清干净”。

### 3. 残留实例识别

残留实例识别分两层：

1. 优先识别 `scripts/dev.mjs` 直接或间接拉起的守护进程
2. 当守护进程不存在时，识别当前单端口派生出的本地端口是否仍被相关进程占用

Windows 下需要以端口反查 PID；Linux/macOS 维持现有守护进程扫描逻辑，并补充端口占用识别。识别结果至少要能给出：

- 发现了哪些残留端口
- 这些端口对应哪些 PID（若可获得）

### 4. 停止行为增强

`bun run dev -- --stop` 的行为改为：

1. 若存在有效守护进程，优先停止守护进程树
2. 若无有效守护进程，但存在残留 PID，则逐个清理残留 PID
3. 若仅发现端口占用但无法归属到 PID，明确向用户报告无法自动清理的端口
4. 若既无守护进程，也无残留端口，才显示“无需停止”

### 5. 文档与模板同步

以下内容需要同步更新：

- `README.md` 中的开发模式说明、默认端口说明、环境变量说明、`--status` / `--stop` 行为说明
- `.env.example` 中的端口配置说明

旧变量 `DEV_WORKER_PORT`、`DEV_ATTEMPT_WORKER_PORT`、`DEV_UI_PORT` 不再出现在文档模板中。

## 兼容性策略

- 脚本实现层面不再读取旧端口变量。
- 现有未设置端口变量的用户无感升级，继续默认走 `8787/8788/4173`。
- 已手工配置旧变量的用户需要迁移到 `DEV_PORT`；README 中明确说明这一点。

## 测试策略

需要新增或扩展自动化测试，覆盖：

1. `DEV_PORT` 到各服务端口的派生规则
2. 后台状态分类新增 `residual`
3. 残留端口存在时，状态输出不再误报“未运行”
4. 停止逻辑在“仅残留实例存在”时会尝试清理

由于本次是脚本行为修复，主要以 `tests/scripts` 下的单元测试覆盖辅助函数与状态判定逻辑；最终再通过实际命令验证：

- `bun run dev -- --status`
- `bun run dev -- --stop`

## 风险与边界

- 端口占用并不一定来自本项目，因此端口反查 PID 后仍需尽量结合命令行特征判断，避免误杀无关进程。
- Windows 与类 Unix 的进程枚举方式不同，平台分支需要保持最小差异且可测试。
- 若日志关闭且守护进程异常退出，状态文件与实际进程之间仍可能短暂不一致，因此状态输出应偏向“暴露异常”而不是“静默当作没事”。
